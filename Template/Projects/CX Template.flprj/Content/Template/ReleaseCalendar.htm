System & Release Maintenance

These are the ForeSee guidelines for rolling out updates. Please be aware that situations arise from time to time which require us change or detour from these guidelines in order to provide the best service to our clients.

If you are interested in viewing recent updates, please see the Release Notes section below.

System Maintenance

System Maintenance pertains to the overall Information Technology of the company and not necessarily any one product or application.

Major Maintenance

The purpose of Major System Maintenance is to update the company's technical infrastructure. These updates are performed in order to implement the latest security measures, optimal performance and reliability, and to support the growth of the ForeSee staff.

Major System Maintenance takes place on the second Saturday of each quarter. This avoids disruptions of the system during the regular work week, along with the end-of-month report preparation, which is standard in our industry. This schedule also complements the Product Release Schedule when Quality Assurance testing and subsequent fixes require a stable environment.

While we make every effort to avoid them, issues do arise requiring an urgent need to implement updates. The Emergency Release process, which is described below, has been developed for these occurrences.

Minor Maintenance

Minor System Maintenance consists of updates that can be implemented without downtime and take place as needed. The process for Minor System Maintenance is described in the Minor Upgrade Approval Form.

Release Maintenance

Release Maintenance, also known as Releases, consists of updates to specific products.

Major Release

New functionality and features for each product are rolled out three times each year:

  • Winter Release - During the first quarter of the calendar year.
  • Spring Release - During the second quarter of the calendar year.
  • Fall Release - At the beginning of the fourth quarter of the calendar year.

There are two important milestones in preparation of Major Release Maintenance, the first of which is a Feature Complete date. This is a one month notice period for all communications concerning the release. Examples of these communication vehicles are Public Facing Release Notes and internal notifications to users of the product(s).

The second important date that occurs two weeks prior to the launch date is Code Freeze. This is where the development of features and functionality ceases and Quality Assurance testing is finishing up on last minute changes.

Patch Release

In between Major Releases, any problems that arise and need to be resolved before the next Major Release, are implemented as needed each Thursday. However, some development teams choose to schedule Patch Releases on a monthly basis. This is due to the complexities of the program, which often require extensive research and testing.

Emergency Release

In the event of an urgent need to implement code, an Emergency Release can be performed under the guidelines set forth in the Emergency Approval Form and is subject to an executive approval process.
